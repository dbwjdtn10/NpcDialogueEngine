# NPC Dialogue Engine — 면접 준비 가이드

## 1. 엘리베이터 피치 (30초)

> "LLM 기반 게임 NPC 대화 시스템입니다.
> RAG 파이프라인으로 세계관 지식을 검색하고, 감정 상태 머신과 호감도 시스템으로
> NPC가 살아있는 것처럼 반응합니다.
> 프롬프트 인젝션 3단계 방어, 퀘스트 스포일러 방지, 크로스-NPC 기억 공유까지
> 게임 대화에 필요한 전체 파이프라인을 설계하고 구현했습니다."

## 2. 프로젝트 소개 (2분 버전)

### 문제 정의
기존 게임 NPC 대화는 스크립트 기반이라 반복적이고 몰입감이 떨어집니다.
LLM을 쓰면 자연스럽지만, 세계관 무시, 캐릭터 이탈, 프롬프트 인젝션, 스포일러 노출 등의 문제가 생깁니다.

### 해결 방법
**8단계 대화 파이프라인**을 설계해서 각 문제를 구조적으로 해결했습니다:
1. 보안 필터 → 2. 의도 분류 → 3. RAG 검색 → 4. 감정 업데이트
5. 응답 생성 → 6. 페르소나 검증 → 7. 호감도 업데이트 → 8. 퀘스트 트리거

### 핵심 기술 선택
- **Hybrid RAG**: Vector(70%) + BM25(30%) + Cross-Encoder 리랭킹
- **한국어 최적화**: Kiwi 형태소 분석기로 BM25 토큰화
- **실시간 스트리밍**: WebSocket 기반 글자 단위 스트리밍

### 인프라
Rate Limiting, Circuit Breaker, Prometheus 메트릭, Structured Logging,
GitHub Actions CI/CD까지 프로덕션 수준의 운영 인프라를 갖추고 있습니다.

## 3. 아키텍처 심화 질문 & 답변

### Q1. "RAG에서 Vector Search만 쓰지 않고 왜 Hybrid를 선택했나요?"

**답변:**
Vector Search는 의미적 유사성은 잘 잡지만, 고유명사나 아이템 이름 같은
**정확한 키워드 매칭에 약합니다.** 예를 들어 "미스릴 검"을 검색할 때
Vector만으로는 "미스릴"이라는 단어가 정확히 포함된 문서를 놓칠 수 있습니다.

BM25를 30% 비중으로 결합하면 키워드 정확도를 보완하면서도,
의미 검색의 장점을 유지할 수 있습니다.

**기술 상세:**
- candidate_k = top_k * 3으로 각각에서 후보를 넓게 뽑고
- content[:200]으로 중복 제거 후 가중 합산
- Cross-Encoder(ms-marco-MiniLM)로 최종 리랭킹하여 정밀도를 높입니다

**수치 근거:** RAG Precision ≥ 0.80, Recall ≥ 0.70 목표를 달성합니다.

### Q2. "감정 시스템은 왜 Rule-based로 했나요? LLM으로 감정을 판단하면 더 자연스럽지 않을까요?"

**답변:**
두 가지 이유입니다.

첫째, **비용**입니다. 이미 Intent Classification에 LLM 1회, Response Generation에 LLM 1회로
턴당 2번의 LLM 호출이 발생합니다. 감정까지 LLM에 맡기면 3번이 되어 레이턴시와 비용이
1.5배가 됩니다.

둘째, **결정론적 제어**입니다. 게임에서 감정 → 호감도 → 콘텐츠 잠금해제로 이어지는
체인이 있는데, LLM의 비결정적 출력에 의존하면 게임 밸런스를 제어할 수 없습니다.

그래서 Intent Classification의 sentiment/intensity를 입력으로 받아
**확정적인 상태 전이 규칙**으로 감정을 결정합니다.

**예시:**
- sentiment=positive, intensity>0.6, intent=quest_inquiry → EXCITED (+15 호감도)
- sentiment=negative, intensity>0.8 → ANGRY (-20 호감도)

### Q3. "Cooldown 메커니즘은 왜 필요한가요?"

**답변:**
플레이어가 같은 칭찬을 반복해서 호감도를 무한히 올리는 **exploit을 방지**하기 위해서입니다.

같은 트리거(예: positive_general_chat)가 3턴 이내에 반복되면:
- 1회 반복: 호감도 변화량 × 0.5 (50% 감소)
- 2회 이상 반복: 호감도 변화량 × 0.0 (완전 무효화)

이렇게 하면 자연스러운 대화에서는 정상적으로 호감도가 변하지만,
기계적 반복은 효과가 사라집니다.

### Q4. "프롬프트 인젝션 방어를 3단계로 나눈 이유는?"

**답변:**
**비용 효율성**과 **방어 깊이** 두 가지입니다.

| 단계 | 방식 | 비용 | 속도 | 역할 |
|------|------|------|------|------|
| 1단계 | Regex 30+패턴 | 무료 | <1ms | 알려진 패턴 즉시 차단 |
| 2단계 | LLM 분류 | API 비용 | ~200ms | 우회 시도 탐지 |
| 3단계 | 출력 검증 | 무료 | <1ms | 메타정보 누출 차단 |

1단계에서 90%+를 잡으면 2단계 LLM 호출이 줄어들어 비용이 절감됩니다.
3단계는 LLM이 실수로 "나는 AI입니다" 같은 메타정보를 말하는 것을 최종 방어합니다.

**Defense in Depth 원칙**: 어느 한 단계가 뚫려도 다음 단계에서 잡습니다.

### Q5. "퀘스트 스포일러 방지는 어떻게 동작하나요?"

**답변:**
퀘스트 문서를 **단계별(stage)로 청킹**합니다.
"전설의 검" 퀘스트면 1단계(의뢰 수락), 2단계(재료 수집), 3단계(제작), 4단계(완성)
각각이 별도 청크가 됩니다.

RAG 검색 시 현재 진행률에 따라 **접근 가능한 단계만 필터링**합니다:
- 0%: max_stage=0 (개요만, "분위기 힌트")
- 30%: max_stage=1 ("기본 방향")
- 60%: max_stage=2 ("구체적 위치/방법")
- 100%: max_stage=3 ("상세 가이드")

이러면 플레이어가 "최종 보스 약점 알려줘"라고 해도,
해당 단계 청크가 검색 대상에서 아예 빠지므로 LLM이 스포일러를 생성할 수 없습니다.

### Q6. "왜 Gemini를 선택했나요? GPT-4나 Claude는?"

**답변:**
세 가지 이유입니다:
1. **한국어 성능**: Gemini 2.0 Flash는 한국어 생성 품질이 우수하면서도 빠릅니다
2. **비용**: Flash 모델은 입력 100만 토큰 기준 GPT-4 대비 약 10배 저렴합니다
3. **레이턴시**: 턴당 LLM 2회 호출이므로 속도가 중요한데, Flash가 응답 속도가 빠릅니다

LangChain을 통해 추상화했으므로 모델 교체는 설정 변경만으로 가능합니다.

### Q7. "Circuit Breaker 패턴을 왜 도입했나요?"

**답변:**
LLM API(Gemini)가 일시적으로 장애가 나면, 모든 사용자 요청이 타임아웃까지
대기하면서 **연쇄 장애(cascading failure)**가 발생합니다.

Circuit Breaker는 연속 실패가 threshold(5회)를 넘으면 회로를 열어서
이후 요청을 **즉시 실패**시킵니다. 이러면:
- 사용자는 빠르게 폴백 응답을 받고 (NPC 캐릭터에 맞는 대사)
- LLM 서비스에 불필요한 부하를 주지 않으며
- recovery_timeout(60초) 후 자동으로 프로브 요청을 보내 복구를 감지합니다

Redis(15초), PostgreSQL(30초), LLM(60초) 각각 다른 타임아웃을 설정한 이유는
서비스별 복구 특성이 다르기 때문입니다.

### Q8. "Sliding Window Rate Limiter를 직접 구현한 이유는?"

**답변:**
slowapi 같은 라이브러리를 쓸 수도 있지만, **의존성을 최소화**하고
알고리즘 이해도를 보여주기 위해 직접 구현했습니다.

Fixed Window 대신 Sliding Window를 선택한 이유는
Fixed Window의 **경계 문제** 때문입니다.
예를 들어 분당 60회 제한에서 59초에 60번, 61초에 60번이면
Fixed Window에서는 2초 동안 120번이 통과하지만,
Sliding Window에서는 윈도우가 연속적으로 이동하므로 이 문제가 없습니다.

수평 확장 시에는 Redis 기반 구현으로 교체하면 됩니다.
그 점도 코드 주석에 명시해두었습니다.

---

### Q9. "메모리 시스템을 왜 2-tier(Redis + PostgreSQL)로 나눴나요?"

**답변:**
게임 대화의 메모리 특성이 다르기 때문입니다.

| | 단기 기억 (Redis) | 장기 기억 (PostgreSQL) |
|---|---|---|
| **내용** | 최근 20턴 원문 | 세션 요약 (3-5문장) |
| **TTL** | 30분 자동 만료 | 영구 보존 |
| **용도** | 문맥 유지, 반복 질문 탐지 | 크로스 세션 기억 |
| **접근** | 매 턴마다 | 세션 시작 시 1회 |

최근 대화는 빠르게 접근해야 하므로 Redis의 O(1) 읽기가 적합하고,
장기 기억은 LLM으로 요약한 후 PostgreSQL에 저장합니다.

**크로스-NPC 기억**: 대장장이 가론과의 대화 요약을 마녀 엘라라가 참조할 수 있어서
NPC간 일관된 세계관을 유지합니다.

**답변 예시 (본인 경험에 맞게 조정하세요):**

> "가장 어려웠던 건 **RAG 검색 품질과 응답 속도의 균형**이었습니다.
> 처음에는 Vector Search만 썼는데 한국어 고유명사 검색이 부정확했습니다.
> BM25를 추가하니 정확도는 올랐지만 레이턴시가 늘었고,
> 결국 Hybrid Fusion + Cross-Encoder 리랭킹으로

> 또 하나는 **스포일러 방지 설계**였습니다.

> 검색 단계에서 아예 미래 단계의 청크를 제거하는 구조적 해결책을 설계했습니다."

## 4. 인프라/DevOps 질문

### Q11. "Prometheus 메트릭을 외부 라이브러리 없이 구현한 이유는?"

> 배포 환경의 의존성을 최소화하기 위해서입니다.
> prometheus_client 라이브러리가 표준이지만, 이 프로젝트에서 필요한 건
> counter, histogram, gauge 3가지뿐이라 직접 구현하는 게 더 가볍습니다.
> 프로덕션 규모에서는 라이브러리를 쓰겠지만, 포트폴리오에서는
> 내부 동작을 이해하고 있다는 것을 보여주고 싶었습니다.

### Q12. "CI/CD 파이프라인은 어떻게 구성되어 있나요?"

> GitHub Actions에 3개 Job이 있습니다.
> 1. **Lint**: ruff로 코드 스타일/정적 분석
> 2. **Test**: pytest로 단위 테스트 (외부 서비스 모킹)
> 3. **Docker Build**: 이미지 빌드 검증 + GHA cache로 빌드 시간 단축
>
> Lint이 먼저 통과해야 Test와 Docker Build가 병렬로 실행됩니다.
> 이건 빠른 피드백(Lint 실패 시 즉시 중단)과 효율성(병렬 실행)의 균형입니다.

### Q13. "Alembic을 도입한 이유는? create_tables()로 충분하지 않나요?"

> `create_tables()`는 테이블이 없으면 새로 만들지만,
> **스키마 변경(컬럼 추가/타입 변경 등)은 처리하지 못합니다.**
> 프로덕션에서 데이터가 있는 상태에서 스키마를 바꿔야 할 때
> Alembic 마이그레이션이 필수적입니다.
> 또한 마이그레이션 히스토리가 코드로 남아서 버전 관리가 됩니다.

## 5. 약점 & 솔직한 대응

### "현재 시스템의 한계는 무엇인가요?"

솔직하게 인정하면서 개선 방향을 제시하면 좋습니다:

1. **수평 확장 미검증**
   - Rate Limiter가 in-memory라서 다중 인스턴스에서는 Redis 기반으로 교체 필요
   - 이 점은 코드 주석에 명시해두었고, 인터페이스가 동일하므로 교체 비용이 낮습니다

2. **LLM 의존도**
   - 턴당 LLM 2회 호출 → API 비용과 레이턴시
   - 개선 방향: Semantic Cache(유사 질문 캐싱, threshold 0.95)로 호출 절감

3. **평가 자동화 부족**
   - 현재는 수동 evaluation dataset(50+ QA쌍)으로 측정
   - 개선 방향: CI에 regression test로 RAG 품질 자동 검증

4. **프론트엔드**
   - 기능 위주 구현, UI/UX 디자인은 기본 수준
   - 백엔드 아키텍처에 집중한 프로젝트

## 6. 기술 스택 선택 근거 요약

| 기술 | 선택 이유 | 대안 대비 장점 |
|------|-----------|---------------|
| **FastAPI** | async 네이티브, WebSocket 지원, 자동 OpenAPI | Django(동기), Flask(수동 async) |
| **Gemini Flash** | 한국어 품질, 저비용, 저레이턴시 | GPT-4(고비용), Claude(API 제한) |
| **ChromaDB** | 로컬 영속화, 임베딩 직접 관리 | Pinecone(클라우드 의존), FAISS(영속화 없음) |
| **multilingual-e5-large** | 한국어 포함 다국어, 높은 검색 정확도 | ko-sroberta(한국어 전용, 범위 제한) |
| **Kiwi** | 한국어 형태소 분석 최고 성능 | Mecab(설치 복잡), Okt(정확도 낮음) |
| **Redis** | 초저지연 캐시, TTL 자동 만료 | Memcached(데이터구조 제한) |
| **PostgreSQL** | JSON 지원, 확장성, 비동기 드라이버 | MySQL(JSON 약함), SQLite(동시성 제한) |

## 8. 면접관 유형별 전략

### 백엔드 면접관
- 미들웨어 스택 실행 순서 설명 (6단계)
- Circuit Breaker 상태 머신 (CLOSED→OPEN→HALF_OPEN)
- Sliding Window vs Fixed Window 차이
- 비동기 처리 (asyncpg, aioredis, WebSocket)

### AI/ML 면접관
- Hybrid RAG 설계 근거 (Vector + BM25 + Cross-Encoder)
- 임베딩 모델 선택 (multilingual-e5-large)
- 한국어 형태소 분석의 BM25 기여도
- Temperature 전략 (0.0 / 0.3 / 0.7 용도별)

## 9. 꼬리 질문 대비

### "이 프로젝트를 확장한다면?"
> - 멀티 유저 동시 접속: WebSocket 세션 관리 + Redis pub/sub
> - NPC 간 실시간 대화: Agent-to-Agent 통신 파이프라인
> - 음성 입출력: Whisper + TTS 연동 (WebSocket 스트리밍 인프라 재활용)
> - A/B 테스트: 다른 LLM 모델 비교 (LangChain 추상화 덕분에 교체 용이)

### "테스트 전략은?"
> - 단위 테스트: 감정 상태 전이, 호감도 경계값, 인젝션 패턴 30+개
> - 통합 테스트: API 엔드포인트, WebSocket 프로토콜
> - 품질 평가: 50+ QA 데이터셋으로 RAG precision/recall 측정
> - 보안 테스트: 직접 탈옥, 간접 탈옥, 페르소나 하이잭, 정보 추출 4개 카테고리

### "왜 이 프로젝트를 만들었나요?"
> (본인의 동기를 자연스럽게 말하되, 아래 포인트를 포함)
> - LLM을 단순 챗봇이 아닌 **게임 시스템에 통합**하는 엔지니어링 도전
> - RAG, 상태 머신, 보안, 인프라를 **하나의 파이프라인에 통합**하는 설계 능력
> - 프로덕션 수준의 **운영 인프라**까지 직접 구축하는 풀스택 역량
