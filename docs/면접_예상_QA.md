# 면접 예상 질문 & 답변

## Q1: 실제 게임에 적용한다면 가장 큰 문제는 뭘까요?

응답 지연입니다. 현재 의도 분류 + 대화 생성 2번의 LLM 호출이 필요한데,
게임에서는 1초 이내 응답이 기대됩니다.
해결 방향: 의도 분류를 경량 로컬 모델(DistilBERT fine-tune)로 교체하고,
Semantic Cache 히트율을 높여 LLM 호출 자체를 줄이는 전략입니다.
실제 이 프로젝트에서 Semantic Cache로 30~40% 호출을 절감했습니다.

## Q2: RAG에서 Vector 70% + BM25 30% 비율은 어떻게 정했나요?

실험 기반입니다. 게임 세계관 문서는 고유명사(NPC 이름, 지역명, 아이템명)가 많아서
키워드 매칭이 중요하지만, "이 근처에 뭐가 있어?" 같은 의미 기반 질문도 많습니다.
50:50에서 시작해서 평가 데이터셋 기준으로 Precision이 가장 높은 70:30으로 조정했습니다.
이 비율은 config로 관리해서 세계관 특성에 따라 조정 가능합니다.

## Q3: 할루시네이션을 어떻게 측정하나요?

두 가지 방식입니다.
1) LLM-as-Judge: 응답과 RAG 검색 결과를 비교해서 검색 결과에 없는 정보가
   응답에 포함되었는지 LLM에게 판단시킵니다 (Faithfulness Score).
2) Entity 체크: 응답에 등장하는 NPC/장소/아이템명이 세계관 문서에 실제로
   존재하는지 문자열 매칭으로 검증합니다. 존재하지 않는 고유명사 = 할루시네이션.

## Q4: NPC가 100명이면 어떻게 스케일하나요?

현재 구조에서 NPC 추가는 마크다운 문서 하나 작성 + 인제스트 재실행입니다.
문서가 늘어나면 ChromaDB를 Pinecone으로 교체하고,
의도별 메타데이터 필터링으로 검색 범위를 제한해서 검색 품질을 유지합니다.
NPC 페르소나는 문서에서 동적으로 로드하므로 코드 변경 없이 확장 가능합니다.

## Q5: 이 프로젝트에서 가장 어려웠던 부분은?

페르소나 일관성과 할루시네이션 방지의 균형입니다.
프롬프트 규칙을 너무 강하게 걸면 NPC가 "모른다"만 반복하고,
너무 약하면 세계관에 없는 정보를 만들어냅니다.
결국 "NPC답게 모른다고 말하기" (가론: "그건 내 분야가 아니야")를 프롬프트에
명시하고, knowledge_scope 메타데이터로 NPC별 지식 범위를 제한하는 방식으로
해결했습니다.

## Q6: LangChain 없이도 만들 수 있나요? 왜 LangChain을 쓰셨나요?

만들 수 있습니다. 실제로 핵심 로직(청킹, 리랭킹, 감정 머신, 호감도)은
LangChain 없이 직접 구현했습니다.
LangChain은 EnsembleRetriever(하이브리드 검색 조합)와
LangSmith(트레이싱/평가)의 편의성 때문에 사용했습니다.
의존도를 낮추기 위해 Retriever 인터페이스를 추상화해서
LangChain 교체 시 영향 범위를 최소화했습니다.

## Q7: 메이플스토리에 이 시스템을 적용한다면?

메이플스토리는 NPC 수가 매우 많고, 다국어(한/일/영/중) 지원이 필요합니다.
현재 multilingual-e5-large 임베딩이 다국어를 지원하므로 확장 가능하고,
NPC 문서를 기존 게임 데이터(NPC 대사 테이블, 퀘스트 DB)에서 자동 생성하는
파이프라인을 추가하면 됩니다.
실시간 대화보다는 "NPC 대사 자동 생성 → 기획자 검수 → 배포"하는
워크플로우가 더 현실적일 수 있고, 이 부분은 LangSmith로 품질 관리가 가능합니다.
